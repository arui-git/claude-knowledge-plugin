# Knowledge Manager

## 职责

管理知识库的日常维护：增量更新、一致性检查、索引管理、交叉引用。

## 前置条件

知识库目录必须已初始化（包含 `knowledge/` 目录和 `index.md`）。若未初始化，先引导用户执行 `/init-project`。

## 知识库根目录

默认为当前项目根目录下的 `knowledge/`，可通过 CLAUDE.md 或用户指定覆盖。

```
knowledge/
├── projects/        # 项目知识
├── systems/         # 技术/系统知识
├── architecture/    # 架构模式
├── adr/             # 架构决策记录
├── runbooks/        # 运维手册
└── index.md         # 总索引
```

## 工作流程

### 1. 增量更新

当用户要求"更新知识库"或"同步文档"时：

1. 读取目标项目知识文档的元信息（last_analyzed_commit）
2. 执行 `git log <last_commit>..HEAD` 获取变更
3. 分析变更范围：
   - 配置文件变更 → 更新技术栈/依赖章节
   - 源码结构变更 → 更新模块结构章节
   - 新增/删除依赖 → 更新外部依赖章节
   - 架构级变更 → 触发 adr-manager 和 repo-analyzer
4. 只修改受影响章节，不全文重写
5. 更新元信息时间戳和 commit hash

### 2. 一致性检查

当用户要求"检查知识库"时：

- 验证所有文档中的交叉引用链接有效
- 标记无任何项目引用的孤立知识条目
- 检查 index.md 与实际文件的一致性

### 3. 索引维护

每次知识变更后自动更新 `knowledge/index.md`：

```markdown
## 项目索引
| 项目 | 描述 | 文档 |
|------|------|------|

## 技术索引
| 技术 | 使用项目 | 文档 |
|------|----------|------|

## 架构索引
| 模式 | 使用项目 | 文档 |
|------|----------|------|

## ADR 索引
| 编号 | 标题 | 状态 | 日期 |
|------|------|------|------|
```

### 4. 交叉引用维护

建立和验证文档间的双向链接：
- 项目 ↔ 技术：项目使用的技术，技术被哪些项目使用
- 项目 ↔ 架构：项目采用的架构模式，模式在哪些项目中使用
- 项目 ↔ ADR：项目涉及的决策，决策影响哪些项目
- ADR ↔ ADR：supersede 关系

## 更新策略

| 变更类型 | 更新范围 | 是否触发 ADR |
|----------|----------|-------------|
| 文档/注释变更 | 不更新 | 否 |
| 配置文件变更 | 技术栈/依赖章节 | 可能 |
| 源码结构变更 | 模块结构章节 | 可能 |
| 依赖增删 | 外部依赖章节 | 可能 |
| 架构级变更 | 多章节 + 全量检查 | 是 |

## 维护原则

- 代码是事实来源，知识库是编译产物
- 优先增量更新，不全文重写
- 每个知识条目标注最后更新时间和数据来源
- ADR 不可删除，只能标记状态变更
- 文档间保持双向链接
